<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cron callback → DB | The cron runs with no authenticated user; it writes via executarGeracaoMensal directly. No user input crosses this boundary — competência is derived internally via competenciaAtual() (date-fns format), never from request data |
| instrumentation register → process runtime | Boot-time side effect executed once per Node process start |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-02 | Tampering | createMany idempotent write | mitigate | DB `@@unique` constraint (Plan 01) + `skipDuplicates: true` make the write idempotent and atomic, surviving a concurrent cron+manual trigger at 06:00 day-1 with no application-level pre-check (no TOCTOU) |
| T-3-03 | Denial of Service / availability | cron job silently failing with no operator visibility | accept | NOTF-01 (email/WhatsApp alerts) is explicitly v2-deferred; v1 mitigation is the `console.log`/`console.error` summary in the callback plus the DONO manual-trigger fallback (D-08, built in Plan 03) |
| T-3-04 | Tampering | double cron registration on hot-reload / Railway redeploy | mitigate | Two independent layers: `globalThis.__agendaFiscalCronStarted` guard (Pitfall 3) AND the DB unique constraint catching any duplicate rows if the guard is ever bypassed |
</threat_model>
