<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → API (login) | credenciais não confiáveis cruzam aqui; validadas server-side |
| browser → rotas autenticadas | cookie de sessão JWT verificado pelo middleware |
| app → Usuario.senhaHash | hash nunca deve sair de authorize() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-AUTH-SPOOF | Spoofing | authorize() / login | mitigate | bcrypt.compare (timing-safe); mensagem de erro genérica "Email ou senha incorretos." não distingue email inexistente de senha errada (Task 1+2) |
| T-01-AUTH-BRUTE | Spoofing | endpoint de login | accept | Rate limiting completo é v1.x (5 usuários internos, baixo risco); mínimo viável = não vazar "usuário existe" na mensagem (já mitigado). Documentado em RESEARCH.md Security Domain |
| T-01-SESSION | Tampering | cookie de sessão JWT | mitigate | Auth.js v5 assina o cookie com AUTH_SECRET; httpOnly/secure/sameSite padrão; CSRF nativo em signIn/signOut |
| T-01-HASH-LEAK | Information Disclosure | queries de Usuario | mitigate | senhaHash só lido em authorize(); todas as outras queries de Usuario usam select explícito omitindo senhaHash |
| T-01-AUTHZ-PUBLIC | Spoofing | rotas autenticadas | mitigate | middleware.ts protege todas as rotas exceto /login e assets; sem sessão → redirect /login |
</threat_model>
