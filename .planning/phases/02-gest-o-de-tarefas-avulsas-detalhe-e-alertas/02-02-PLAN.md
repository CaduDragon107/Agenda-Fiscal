Implementar `criarTarefa(formData: FormData): Promise<AcaoTarefaResult>`:
- Guard: `const session = await auth(); if (!session?.user) return { ok: false, error: "Não autenticado" };`
- Parse: extrair titulo, descricao, empresaId, responsavelId, prazo do FormData com `formData.get(...)`
- Validar: `tarefaSchema.safeParse({ titulo, descricao, empresaId, responsavelId, prazo })` — se falhar, retornar `{ ok: false, error: "Não foi possível criar a tarefa. Verifique os dados e tente novamente." }`
- Criar: `db.tarefa.create({ data: { titulo, descricao, empresaId, responsavelId, prazo (Date), status: "PENDENTE" } })`
- Revalidar: `revalidatePath("/tarefas")`
- Retornar: `{ ok: true, id: tarefa.id }`

Implementar `concluirTarefa(id: string): Promise<AcaoTarefaResult>`:
- Guard de sessão (idêntico ao padrão)
- Anti-IDOR: `const existente = await db.tarefa.findFirst({ where: { id, ...withTarefaScope(session.user) }, select: { id: true, status: true } }); if (!existente) return { ok: false, error: "não encontrado" };`
- Idempotência: `if (existente.status === "CONCLUIDA") return { ok: true };`
- Transaction: `await db.$transaction([ db.tarefa.update({ where: { id }, data: { status: "CONCLUIDA" } }), db.tarefaHistorico.create({ data: { tarefaId: id, concluidoPorId: session.user.id, concluidoEm: new Date() } }) ])` — per RESEARCH.md Open Question 1 (atomicidade garante que se o create do histórico falhar, o status não fica como CONCLUIDA sem registro)
- Revalidar: `revalidatePath("/tarefas")` e `revalidatePath(\`/tarefas/${id}\`)`
- Retornar: `{ ok: true }`

Implementar `excluirTarefa(id: string): Promise<AcaoTarefaResult>`:
- Guard de sessão
- Anti-IDOR: `findFirst` escopado (igual ao padrão de `excluirEmpresa` em `src/app/(app)/actions.ts`)
- Delete: `db.tarefa.delete({ where: { id } })` — TarefaHistorico é deletado em cascade (onDelete: Cascade no schema)
- Revalidar: `revalidatePath("/tarefas")`
- Retornar: `{ ok: true }`

CRÍTICO (T-02-UNAUTH, T-02-IDOR): Toda action DEVE ter o guard de sessão como primeira instrução. Toda mutation DEVE ter o findFirst escopado ANTES de qualquer write. NUNCA usar `findUnique` em mutations — não aceita scope composto.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| FormData → tarefaSchema.safeParse | Dados não confiáveis chegam do browser via Server Action; Zod é a barreira de validação |
| session.user.id → withTarefaScope | O id do usuário da sessão determina quais tarefas são visíveis/mutáveis; nunca aceitar userId do cliente |
| db.tarefa.findFirst(escopo) → db.tarefa.update | O resultado do findFirst determina se a mutation prossegue; null = acesso negado |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-IDOR | Elevation of Privilege | `concluirTarefa`, `excluirTarefa` | mitigate | `findFirst({ where: { id, ...withTarefaScope(user) } })` antes de qualquer write; retorna "não encontrado" (nunca 403) — testado em `tests/tarefas.idor.test.ts` |
| T-02-INPUT | Tampering | `criarTarefa` — campos titulo, descricao campos livres via FormData | mitigate | `tarefaSchema.safeParse(formData)` valida presença e tipos antes de qualquer `db.tarefa.create`; Prisma usa queries parametrizadas — sem SQL injection possível |
| T-02-UNAUTH | Spoofing | Toda Server Action sem sessão | mitigate | Guard `const session = await auth(); if (!session?.user) return { ok: false, error: "Não autenticado" }` como primeira instrução em criarTarefa, concluirTarefa, excluirTarefa |
| T-02-FK | Tampering | empresaId e responsavelId manipulados para IDs inexistentes | mitigate | `z.string().min(1)` valida presença; Prisma FK constraint rejeita na escrita com erro de banco (não expõe dados, apenas falha) |
| T-02-SC | Tampering | npm install date-fns | accept | date-fns auditado no RESEARCH.md como aprovado (OK): ~9 anos, ~80M downloads/semana, package legítimo; sem interação com dados do usuário |
</threat_model>
